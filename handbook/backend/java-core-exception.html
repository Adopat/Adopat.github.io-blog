    # 第四章 异常

    # 4.1 异常体系

    # 4.2 异常和错误的概念

    • 异常

      通常情况下是可以被程序处理的，捕获后可能恢复，并且在程序中应该尽可能地去处理这些异常。

    • 错误

      通常是灾难性的致命的错误，是程序无法控制和处理的，当出现这些错误时，建议终止程序；

    # 4.3 checked和unchecked异常

    • checked异常(受检异常)

      除了 RuntimeException 及其子类以外，其他的 Exception 类及其子类都属于这种异常。在源代码里必须显式地捕获或者抛出，否则编译器会提示你进行相应的操作

    • unchecked异常(非受检异常)

      包括 RuntimeException 及其子类和 Error。通常是可以通过编码进行规避的，并不需要显式地捕获或者抛出。

    # 4.4 运行时异常和非运行时异常

    • 运行时异常 ：包括RuntimeException 及其子类。比如 NullPointerException、IndexOutOfBoundsException。属于非受检异常，可以进行捕捉处理，也可以不处理。
    • 非运行时异常（编译异常）：RuntimeExcaption 以外的 Exception。IOException、SQLException 已经自定义的异常，必须要进行处理。

    # 4.5 常见的异常分类

    类型 说明

    ClassNotFoundException 和 NoClassDefFoundError 区别

    1. **ClassNotFoundException**是指在运行时，当程序试图加载一个类时，找不到该类的字节码文件或者该类所依赖的类的字节码文件时，抛出的异常。通常情况下，ClassNotFoundException是由于类路径错误或者类名拼写错误导致的。
    2. **NoClassDefFoundError**是指在运行时，当程序试图使用某个类时，该类的字节码文件存在，但是在加载该类的时候，该类所依赖的类或接口无法找到或无法加载时，抛出的异常。通常情况下，NoClassDefFoundError是由于类路径错误或者类版本不一致导致的。
    3. **ClassNotFoundException和NoClassDefFoundError**的主要区别在于抛出异常的时机不同。ClassNotFoundException是在类加载的过程中抛出的异常，而NoClassDefFoundError是在类初始化的过程中抛出的异常。
    4. ClassNotFoundException是一个受检异常，需要在代码中显式地进行处理，而NoClassDefFoundError是一个非受检异常，不需要进行处理。

    # 4.6 关于 throw 和 throws

    1. throws 关键字用于声明异常，它的作用和 try-catch 相似；而 throw 关键字用于显式的抛出异常。
    2. throws 关键字后面跟的是异常的名字；而 throw 关键字后面跟的是异常的对象。
    3. throws 关键字出现在方法签名上，而 throw 关键字出现在方法体里。
    4. throws 关键字在声明异常的时候可以跟多个，用逗号隔开；而 throw 关键字每次只能抛出一个异常。
    5. throw 用于抛出异常 throws用于声明异常。
    // throws 声明异常
throws ArithmeticException;
// throw 抛出异常
throw new ArithmeticException("算术异常");

    # 4.7 关于 try-catch-finally

    try 块：用于捕获异常。其后可接零个或多个 catch 块，如果没有 catch 块，则必须跟一个 finally 块。

    catch 块：用于处理try捕获到的异常。

    finally 块：无论是否捕获或处理异常，finally 块里的语句都会被执行。 当在 try 块或 catch 块中遇到 return 语句时，finally 语句块将在方法返回之前被执行。

    在以下 4 种特殊情况下，finally 语句块不会被执行：

    • 在finally语句块中发生了异常
    • 在前面的代码中用了System.exit()退出程序。
    • 程序所在的线程死亡。
    • 关闭 CPU。

    # 4.8 自定义异常

    • 自定义异常使用步骤

      1. 继承Exception或RuntimeException类，或者继承其他自定义异常类，从而实现异常类的层次结构。
      2. 在异常类中定义构造方法，可以传入错误代码、错误信息等参数。
      3. 在代码中使用throw语句抛出自定义异常。
      4. 在调用代码中使用try-catch语句捕获自定义异常，并处理异常。

    • 实例代码

      // 1.继承Exception或RuntimeException类，或者继承其他自定义异常类，从而实现异常类的层次结构。
public class MyException extends Exception {
    private int errorCode;
    // 2.在异常类中定义构造方法，可以传入错误代码、错误信息等参数。
    public MyException(int errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    public int getErrorCode() {
        return errorCode;
    }
}

// 3.在代码中使用throw语句抛出自定义异常。
public class MyClass {
    public void doSomething() throws MyException {
        // 业务逻辑
        if (somethingWrong) {
            throw new MyException(1001, "Something went wrong");
        }
        // 业务逻辑
    }
}

// 4.在调用代码中使用try-catch语句捕获自定义异常，并处理异常。
public class MainClass {
    public static void main(String[] args) {
        MyClass myClass = new MyClass();
        try {
            myClass.doSomething();
        } catch (MyException e) {
            System.out.println("Error code: " + e.getErrorCode());
            System.out.println("Error message: " + e.getMessage());
            // 处理异常
        }
    }
}

    # 4.9 try-with-resouces语法糖

    • 概念

      Java 7引入了try-with-resources语法糖，它可以在try语句中自动关闭资源，无需手动关闭。

    • 语法格式

      try (resource-1; resource-2; ... resource-n) {
    // use resources
} catch (Exception e) {
    // handle exceptions
}
      1
      2
      3
      4
      5

      其中，resource可以是实现了AutoCloseable接口的任何对象，包括IO流、数据库连接、网络连接等等。

    • 作用

      使用try-with-resources语法糖可以自动关闭资源，避免因为忘记关闭资源而导致的资源泄漏问题。同时，try-with-resources还可以自动处理多个资源的关闭顺序，保证所有资源都能正确关闭。

    • 原理

      try-with-resources语法糖的原理是使用try语句块中的变量自动调用close()方法。在try语句结束时，系统会自动调用close()方法关闭资源。如果在执行close()方法时抛出异常，那么这个异常将被抑制，而不是覆盖原来的异常。

    • 注意事项

      • 资源必须实现AutoCloseable接口，否则无法使用try-with-resources语法糖。

      • 可以在try语句中使用多个资源，用分号隔开。

      • 在try语句中定义的资源变量必须是final或者是effectively final，否则会导致编译错误。

        所谓final变量，就是变量只能被赋值一次，且在生命周期内不能被改变。而effectively final变量，是指虽然没有使用final关键字声明，但是在变量的生命周期内，变量的值不会被改变，也就相当于是一个final变量。

        effectively final变量是指虽然没有使用final关键字声明，但是在变量的生命周期内，变量的值不会被改变，也就相当于是一个final变量。

        Java 8之前，如果在匿名内部类或lambda表达式中使用局部变量，那么这个变量必须是final变量。这是因为Java 8之前的版本中，匿名内部类和lambda表达式共享的是变量的副本，而不是变量本身。如果变量不是final的，那么在匿名内部类或lambda表达式中改变变量的值，会导致编译错误或运行时异常。

        Java 8之后，可以使用effectively final变量来代替final变量。如果变量在定义后没有被重新赋值，那么它就是effectively final变量，可以在匿名内部类和lambda表达式中使用，而不需要显式地声明为final变量。

        例如，下面的代码中，i是effectively final变量：

        int i = 0;
Runnable r = new Runnable() {
    @Override
    public void run() {
        System.out.println(i); // 可以访问effectively final变量i
    }
};

        在上面的代码中，我们定义了一个Runnable对象r，可以在其中访问外部变量i。由于i没有被重新赋值，所以它是effectively final变量，可以在匿名内部类中使用。

        使用effectively final变量可以使代码更加简洁，避免了显式声明final变量的麻烦。但需要注意的是，如果变量在定义后被重新赋值，那么它就不再是effectively final变量，需要显式声明为final变量才能在匿名内部类和lambda表达式中使用。

        这个规定是为了保证资源变量在try语句块中的生命周期内不会被改变，从而确保资源能够被正确关闭。如果资源变量不是final或者effectively final，那么在try语句块中可能会改变变量的值，从而导致关闭错误的资源或者无法关闭资源的情况。

      • 在catch语句中处理异常时，需要注意是否有抑制异常，可以通过Throwable.getSuppressed()方法获取抑制异常。

        在Java中，当一个异常被抛出时，可能会伴随着其他异常的抛出。这些附加的异常被称为抑制异常。

        抑制异常是指在处理一个异常时，可能会发生其他异常，而这些异常被抑制了，即被压制在了主要异常之下，不会被立即抛出，而是在主要异常被处理后被抛出。

        Java 7引入了try-with-resources语法糖，它可以自动关闭资源，同时也可以处理抑制异常。在try-with-resources语法糖中，如果在关闭资源时发生异常，那么这个异常会被抑制，而不是覆盖原来的异常。可以通过Throwable.getSuppressed()方法获取抑制异常。

        例如，下面的代码中，try-with-resources语句块中的close()方法可能会抛出异常，这个异常被抑制了：

        try (FileInputStream fis = new FileInputStream("file.txt")) {
    // do something
} catch (IOException e) {
    // handle exception
    Throwable[] suppressed = e.getSuppressed();
    for (Throwable t : suppressed) {
        System.out.println("Suppressed exception: " + t);
    }
}

        需要注意的是，抑制异常只有在Java 7及以上版本中才支持，如果在Java 6及以下版本中使用try-with-resources语法糖，抑制异常会被忽略。

      不使用 try-with-resources语法

      import java.io.*;

/**
 * IO 流中异常的处理 JDK6以前的处理方法 在finally 中做多次判断进行流的关闭
 */
public class TryCatchTest {
    public static void main(String[] args){
        BufferedInputStream bis = null;
        BufferedOutputStream bos = null;
        try{
            InputStream is = new FileInputStream("C:\\myNote\\test_20210907\\1\\P002.jpg");
            bis = new BufferedInputStream(is);
            OutputStream os = new FileOutputStream("C:\\myNote\\test_20210907\\1\\COPY_20210913_P002.jpg");
            bos = new BufferedOutputStream(os);
            int size;
            byte[] buff = new byte[1024];
            while( (size = bis.read(buff))!=-1){
                bos.write(buff,0,size);
            }
        }catch (Exception e){
                e.printStackTrace();
        }finally {
            if(bis!=null){
                try{
                    bis.close();
                }catch (IOException e){
                    e.printStackTrace();
                }finally {
                    if(bos!=null){
                        try{
                            bos.close();
                        }catch (IOException e){
                            e.printStackTrace();
                        }
                    }
                }
            }
        }

    }
}
